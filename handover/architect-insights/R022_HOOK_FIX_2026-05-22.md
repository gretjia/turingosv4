# R-022 commit-hook footgun fix — pre-commit → commit-msg

**Date**: 2026-05-22
**Risk class**: 1 (additive harness fix on tooling)
**FC-trace**: FC3-Alignment (TRACE_MATRIX_v3 backlink discipline)
**Predecessor**: `handover/research/PLAN_V7_MINIF2F_RECOVERY_2026-05-22/C_HOTFIX_R1_LEAK_POSTMORTEM.md` §4
**Related**: `handover/alignment/OBS_R022_R1_EXTERNAL_MARKET_SNAPSHOT_LEAK_2026-05-22.md`

## 1. The footgun (latent since CO1.13.2)

The R-022 trace-matrix backlink check originally ran from a **pre-commit**
hook that read the in-flight commit message from `.git/COMMIT_EDITMSG`.

Behavior of `COMMIT_EDITMSG` per git internals:

| invocation | COMMIT_EDITMSG state at pre-commit-hook time |
|---|---|
| `git commit` (editor) | populated (editor writes through it) |
| `git commit -m "msg"` | **stale (previous commit) — populated AFTER pre-commit passes** |
| `git commit -F file`  | **stale (previous commit) — populated AFTER pre-commit passes** |

The R-022 skip token `[R-022-skip: <ref>]` lives in the new commit's
message. With `-m` or `-F`, the pre-commit hook saw the PREVIOUS commit's
message, which obviously didn't carry the new commit's token — so the
removal block fired even when a valid token was present.

Reproduction (2026-05-22 R1 hotfix attempt):

```bash
# Token validator says message is valid:
GIT_COMMIT_MSG="$(cat /tmp/r022_hotfix_msg.txt)" \
  python3 scripts/check_trace_matrix.py --mode commit
# → exit 0

# But actual git commit blocks:
git commit -F /tmp/r022_hotfix_msg.txt
# → BLOCKED by R-022 (TRACE_MATRIX pub-symbol-block)
```

The ugly workaround used during R1 was to pre-populate `COMMIT_EDITMSG`
manually:

```bash
cp /tmp/r022_hotfix_msg.txt .git/COMMIT_EDITMSG \
  && git commit -F /tmp/r022_hotfix_msg.txt
```

That is not a fix — it's a footgun-in-waiting. Any future automation that
commits via `-m`/`-F` and forgets the `cp` would re-hit the same block.

## 2. The fix

**Move the R-022 check from `pre-commit` to `commit-msg`.** The
`commit-msg` hook is git's standard validation phase for message content
and receives the in-flight message file path as `$1` regardless of how the
commit was triggered (`-m`, `-F`, interactive editor). It runs AFTER
pre-commit and BEFORE the commit object is created, so it preserves the
"block before the commit lands" semantics.

### File changes

| file | change |
|---|---|
| `scripts/check_trace_matrix.py` | Added `--message-file <path>` flag with priority `--message-file > GIT_COMMIT_MSG env > .git/COMMIT_EDITMSG`. Refactored `run()` to lazy-bind `PROJECT_ROOT` for testability. |
| `scripts/hooks/commit-msg.r022` | **NEW.** Tracked shim that passes `$1` as `--message-file`. |
| `scripts/hooks/pre-commit.r022` | Removed the trailing R-022 `exec`. Now does ONLY the K-HARDEN-2 sidecar contamination block (separate concern, unchanged behavior). |
| `scripts/install_hooks.sh` | Installs `commit-msg` symlink alongside `pre-commit` and `pre-push`. Idempotent. |
| `scripts/tests/test_check_trace_matrix_commit_msg.py` | **NEW.** 6 unit + integration tests pinning the fix. |

### Priority chain (commit_message)

```
--message-file <path>     # commit-msg hook ($1)         ← authoritative
  ↓ (not provided)
GIT_COMMIT_MSG env        # test injection / CI overrides
  ↓ (not set)
.git/COMMIT_EDITMSG       # interactive editor fallback (still works)
  ↓ (missing)
""                        # no token → block fires if violations
```

### CI mode unchanged

`--mode ci` still reads `git log -1 --pretty=%B`; that path was never
broken and isn't touched by this fix. (The PR-mode brittleness against
synthetic merge commits — postmortem §5 — remains as a known limitation;
orchestrator continues to rely on main-push CI for ground truth.)

## 4. Why not the alternatives

**Option B — `prepare-commit-msg` hook**: fires earlier and can MUTATE
the message. It's a message-construction phase, not a validation phase.
Using it for blocking would conflate two responsibilities; future
prepare-commit-msg consumers could surprise the validator.

**Option C — require `GIT_COMMIT_MSG` env var on every commit**: bad UX,
fails open if forgotten. Rejected.

**Option A (chosen) — commit-msg hook**: this is the phase git designed
for exactly this purpose. The shim is a 3-line passthrough; the script
gains one flag and one priority rule. Minimal surface area, fail-closed
behavior preserved.

## 5. Migration / install

Existing clones must re-run:

```bash
bash scripts/install_hooks.sh
```

The script is idempotent — it removes the stale `pre-commit` symlink and
adds the new `commit-msg` symlink. No manual `.git/hooks/` cleanup
needed.

CI does not run `install_hooks.sh`; `.github/workflows/co1_13_r022_ci.yml`
keeps calling `--mode ci` directly and is unaffected.
