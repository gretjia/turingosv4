## 0. One-line goal

Close the **first half of the challenge-resolution loop** by adding `ChallengeResolveTx` (system-emitted) with **Released-only** active semantics: challenger's NO bond returns from `challenge_cases_t.bond` to `balances_t[challenger]`, the `ChallengeCase` entry is removed, and CTF round-trips back to the pre-challenge state. The dual outcome (`UpheldDeferred`) is a marker only — slashing is **TB-6 (RSP-3.2)** territory. After TB-5: a challenge can be admitted (TB-4) and dismissed (TB-5); the slash path stays absent. Q_t carries enough provenance — `accepted_at_round` on `StakeEntry` (TB-5 NEW) + `challenge_window_length` on `TaskMarketEntry` (TB-5 NEW) — for **TB-6** to install the deadline math without further schema bumps.

### 3.1 WP-canonical: `ChallengeResolveTx` is a first-class new `TypedTx` variant

Per the WP-vs-Roadmap reconciliation rule (memory `feedback_wp_vs_roadmap_reconciliation`; codified TB-3, re-applied TB-4):

- **Allowed-named variants** (those literally named in WP § 19 / ROADMAP § 3 P3 transaction list): `task_open_tx` (TB-3 ✓), `escrow_lock_tx` (TB-3 ✓), `verify_tx` (TB-4 ✓), `challenge_tx` (TB-4 ✓), **`challenge_resolve_tx` (TB-5 NEW)**, `provisional_accept_tx` (RSP-3.x deferred), `settlement_tx` (RSP-4 deferred), `slash_tx` (RSP-3.2 deferred), `reputation_update_tx` (RSP-3.3 deferred).

The TB-3 / TB-4 anti-drift CI (I44) does **not** forbid this — its forbidden literal list excludes `ChallengeResolveTx` (I44 forbids only the four phantom-variants `NoStakeTx` / `VerifierBondTx` / `ChallengeStakeTx` / `VerifierStakeTx`).

### 3.2 System-emitted, NOT agent-submitted

`ChallengeResolveTx` is a **system-emitted** transaction (mirroring `FinalizeRewardTx` / `TaskExpireTx` / `TerminalSummaryTx`). Its signing payload uses `SystemSignature` + `SystemEpoch` (NOT `AgentSignature`), and its `HasSubmitter::submitter_id()` returns `None` (the L4.E rejection-evidence path falls back to `SYSTEM_AGENT_ID_STR` — TB-2 inheritance).

**Why system-emitted**: a challenge resolution is the constitutional verdict on a posted challenge. Per WP architecture § 12.4 + Constitution Art V.1.3 (JudgeAI guard), only the trusted top-white-box can emit resolutions — agents (Solver / Verifier / Challenger) are objects of resolution, never authors. v4 stays at "離線 dual audit (Codex + Gemini)" as JudgeAI proxy; the system-keypair signs ChallengeResolveTx as the runtime's mechanical encoding of "the configured resolution policy says X."

**Trigger source in TB-5**: TB-5 does NOT install an automatic trigger. Tests submit ChallengeResolveTx manually through `Sequencer::submit` (test fixtures construct + system-sign). Production triggers (operator-emitted; future automatic-tick) are RSP-3.x / P2.

### 3.5 ChallengeResolveTx admission semantics (Released path = TB-5 active)

A `ChallengeResolveTx` is admissible iff:

1. `parent_state_root == q.state_root_t` (StaleParent gate).
2. `target_challenge_tx_id` exists in `q.economic_state_t.challenge_cases_t.0` — else `ChallengeNotFound` (NEW TransitionError variant per § 4.4).
3. (Implicit) the existing ChallengeCase entry has not yet been resolved (no AlreadyResolved gate needed in TB-5 because step 2 + atomic remove means a resolved ChallengeCase IS the absent-from-Q signal — same shape applies through every TB-5 sequence; if RSP-3.2 introduces a non-removal "marked-resolved" state, AlreadyResolved gets added then).
4. `resolution` is one of {Released, UpheldDeferred}.
5. **NO window-closure check** (deferred TB-6 — gate would be `q.q_t.current_round - target_stake.accepted_at_round >= task_market.challenge_window_length` but TB-5 deliberately admits resolutions at any round to keep slash-deferred semantics atomic).
6. **NO solver/verifier solvency check** (TB-5 does not move solver/verifier money).

On accept:

```text
case = q.challenge_cases_t.0.get(&target_challenge_tx_id)  // verified at step 2

match resolution:
  Released:
    balances_t[case.challenger] += case.bond
    challenge_cases_t.0.remove(&target_challenge_tx_id)

  UpheldDeferred:
    // Q_t MUTATIONS = ZERO. ChallengeCase stays put.
    // Marker ride canonical L4 only; semantics for RSP-3.2.
    (no q_next mutation)

state_root_t = sha256(CHALLENGE_RESOLVE_DOMAIN_V1 ‖ prev ‖ canonical_hash(tx))

assert_no_post_init_mint(tx, q)
assert_total_ctf_conserved(q.economic_state_t, q_next.economic_state_t, &[])
  Released:       OK trivially (bond moved sideways; debit challenge_cases.bond
                   = credit balances; CTF closes)
  UpheldDeferred: OK trivially (Q_t bit-identical)
```

**Key CTF property**: TB-5 Released is the **inverse** of TB-4 ChallengeTx accept. TB-4 moved `bond` from `balances_t[challenger]` → `challenge_cases_t[challenge.tx_id].bond`. TB-5 Released moves it back. Round trip → identical pre-challenge `economic_state_t` (modulo the L4 row count which advances both times).

### 3.6 Released DOES NOT auto-release Solver / Verifier stakes

**Critical scope boundary**: a single ChallengeResolveTx{Released} dismisses **one** ChallengeCase. It does NOT:

- Remove the target Solver's `stakes_t[work.tx_id]` entry (Solver YES stake stays locked PROVISIONAL pending RSP-4 FinalizeRewardTx).
- Remove the Verifier's `stakes_t[verify.tx_id]` entry (Verifier bond stays locked pending RSP-4 settlement).
- Decrement `task_markets_t[work.task_id].total_escrow` (escrow stays put for RSP-4 to settle).
- Touch `claims_t` / `reputations_t` / `royalty_graph_t` / `price_index_t`.

This means, after a sequence of N Challenges all resolved Released, the Solver's stake remains locked. That's the **correct constitutional shape**: bond release is per-challenger; Solver stake release is RSP-4's job (FinalizeRewardTx settlement).

### 3.7 NO automatic round-advancement

`q.q_t.current_round` is **NOT mutated** by ChallengeResolveTx (or by any TB-5 dispatch arm). The window-closure math TB-6 will install presupposes a round-advancement signal — that signal is **P2 Agent Runtime territory** (per ROADMAP § 11 dependency graph: P2 "depends on RSP-1; produces role-separation events that include round ticks"). TB-5 just records the post-resolve `current_round` value into the system_signature timing (via `epoch.timestamp_logical`) so RSP-3.2 / TB-6 has a Q-derivable monotonic clock available without requiring a new round-tick TypedTx variant.

For TB-5 tests, `current_round` is set manually in fixtures (mirror TB-4 I43 pattern: `q.q_t.current_round = 7` then assert `opened_at_round == 7`).

### 3.8 5-class TransitionError taxonomy continues; +2 new variants

### 3.9 No `ProvisionalAcceptTx` / `SettlementTx` / `SlashTx`

TB-5 does **NOT** introduce any of:
- `ProvisionalAcceptTx` — provisional vs final transition is implicit via stakes_t entry presence (Solver stake still in stakes_t = provisional). Explicit ProvisionalAcceptTx is a RSP-3.x candidate but unnecessary in TB-5 minimum scope.
- `SettlementTx` — RSP-4 territory.
- `SlashTx` — RSP-3.2 territory.
- `ReputationUpdateTx` — RSP-3.3 / P4 territory.

CI-enforced anti-drift extends TB-4's I44 with a TB-5 sibling test (§ 4.7).

### 3.10 ChallengeResolveTx is signal+resolution, NOT subjective slash decision

**Mirror of TB-4 § 3.10** (VerifyTx is signal+stake, not judge): ChallengeResolveTx is the runtime's **mechanical encoding** of a configured resolution policy. The decision rule (which challenges get Released vs UpheldDeferred) is determined by:

- Whether the counterexample_cid in the corresponding `ChallengeTx` (recoverable from L4) actually produces a counterexample under the predicate registry — but this **predicate evaluation is NOT done in TB-5 dispatch**; it's a future RSP-3.x runtime upgrade.
- For TB-5, **all submitted ChallengeResolveTx with valid system signature accept**. The policy decision is OUT-OF-BAND (operator-driven; tests parameterize over Released vs UpheldDeferred as input).

This is **not a backdoor**: the system-keypair is the constitutional gate (Constitution Art V.1.3 + WP § 12.4); only Architect-AI / RootBox can produce a valid SystemSignature. Agents cannot forge ChallengeResolveTx because they don't hold the system_keypair. So accepting all valid-signed resolutions is safe at this layer.

### 4.5 ChallengeResolveTx dispatch arm — design

Insert as a new arm in sequencer.rs `dispatch_transition` (after the existing 9 arms):

```rust
TypedTx::ChallengeResolve(resolve) => {
    // Step 1: parent-root match.
    if resolve.parent_state_root != q.state_root_t {
        return Err(TransitionError::StaleParent);
    }
    // Step 2: target ChallengeCase exists.
    let case = match q.economic_state_t.challenge_cases_t.0
        .get(&resolve.target_challenge_tx_id) {
        Some(c) => c.clone(),
        None => return Err(TransitionError::ChallengeNotFound),
    };
    // Step 3: build q_next.
    let mut q_next = q.clone();
    match resolve.resolution {
        ChallengeResolution::Released => {
            // Refund challenger.
            let cur = q.economic_state_t.balances_t.0
                .get(&case.challenger).copied()
                .unwrap_or(MicroCoin::zero());
            let new_bal = cur.micro_units() + case.bond.micro_units();
            q_next.economic_state_t.balances_t.0.insert(
                case.challenger.clone(),
                MicroCoin::from_micro_units(new_bal),
            );
            // Remove ChallengeCase.
            q_next.economic_state_t.challenge_cases_t.0
                .remove(&resolve.target_challenge_tx_id);
        }
        ChallengeResolution::UpheldDeferred => {
            // No mutation. Marker only.
        }
    }
    // Step 4: monetary invariants.
    assert_no_post_init_mint(tx, q)
        .map_err(|_| TransitionError::MonetaryInvariantViolation)?;
    assert_total_ctf_conserved(
        &q.economic_state_t,
        &q_next.economic_state_t,
        &[],
    )
    .map_err(|_| TransitionError::MonetaryInvariantViolation)?;
    // Step 5: state_root advance via CHALLENGE_RESOLVE_DOMAIN_V1.
    q_next.state_root_t = challenge_resolve_accept_state_root(&q.state_root_t, tx);

    Ok((q_next, SignalBundle::default()))
}
```
